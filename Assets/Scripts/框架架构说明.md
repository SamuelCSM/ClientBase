# Unity客户端框架架构说明

## 📐 整体架构

本框架采用**分层架构**设计，分为两个主要层级：

```
┌─────────────────────────────────────────────────────┐
│                   游戏应用                            │
├─────────────────────────────────────────────────────┤
│                                                       │
│  ┌──────────────────────────────────────┐           │
│  │   HotUpdate层（热更新层）             │           │
│  │   可热更新 ✅                         │           │
│  ├──────────────────────────────────────┤           │
│  │  • 游戏逻辑代码                       │           │
│  │  • UI逻辑                            │           │
│  │  • 战斗系统                          │           │
│  │  • 配置表定义                        │           │
│  └──────────────────────────────────────┘           │
│                      ↑                                │
│                   调用接口                            │
│                      ↓                                │
│  ┌──────────────────────────────────────┐           │
│  │   Framework层（框架层）               │           │
│  │   不可热更新 ❌                       │           │
│  ├──────────────────────────────────────┤           │
│  │  • 核心管理器（8个Manager）           │           │
│  │  • 资源管理                          │           │
│  │  • 网络通信                          │           │
│  │  • UI管理                            │           │
│  │  • 热更新管理                        │           │
│  └──────────────────────────────────────┘           │
│                                                       │
└─────────────────────────────────────────────────────┘
```

## 🎯 两层架构详解

### 1️⃣ Framework层（AOT层）

#### 📍 位置
```
Assets/Scripts/Framework/
```

#### 🔧 编译方式
- **AOT编译**（Ahead-Of-Time）
- 编译到主程序中，随安装包发布
- 使用IL2CPP或Mono编译

#### ✨ 特点
- ✅ 性能最优（AOT编译）
- ✅ 稳定可靠
- ❌ **不能热更新**（修改需要重新发布安装包）
- ✅ 提供底层服务

### 2️⃣ HotUpdate层（热更新层）

#### 📍 位置
```
Assets/Scripts/HotUpdate/
```

#### 🔧 编译方式
- 编译成 **HotUpdate.dll**
- 通过HybridCLR在运行时加载
- 可以从服务器下载更新

#### ✨ 特点
- ✅ **可以热更新**（无需重新发布安装包）
- ✅ 灵活可变
- ✅ 快速迭代
- ⚠️ 性能略低于AOT（但差异很小）

#### 📦 包含内容
```
HotUpdate/
├── Entry/                  # 热更新入口
│   └── HotfixEntry.cs     # 热更新启动入口
│
├── Logic/                  # 游戏逻辑
│   ├── Player/            # 玩家系统
│   ├── Battle/            # 战斗系统
│   ├── UI/                # UI逻辑
│   └── ...
│
├── Config/                 # 配置表定义
│   └── ConfigDefine/
│
└── Proto/                  # Protobuf协议
    └── *.cs
```

#### 🎯 设计原则
- **业务逻辑** - 所有游戏逻辑都放这里
- **频繁变化** - 经常需要修改的代码
- **依赖Framework** - 调用Framework层提供的服务

## 🔄 两层交互方式

### HotUpdate层调用Framework层

```csharp
// 在HotUpdate层的代码中
using Framework.Core;
using Cysharp.Threading.Tasks;

namespace HotUpdate.Logic
{
    public class PlayerController
    {
        public async void LoadPlayerUI()
        {
            // 调用Framework层的UIManager
            var ui = await GameEntry.UI.OpenUIAsync<PlayerUI>();
            
            // 调用Framework层的ResourceManager
            var prefab = await GameEntry.Resource.LoadAssetAsync<GameObject>("Player/Model");
            
            // 调用Framework层的EventManager
            GameEntry.Event.TriggerEvent(EventDefine.PlayerLogin);
        }
    }
}
```

### Framework层不直接调用HotUpdate层

Framework层通过**反射**启动HotUpdate层：

```csharp
// Framework层的HotUpdateManager
public void StartHotfix()
{
    // 通过反射创建HotfixEntry实例
    Type entryType = _hotUpdateAssembly.GetType("HotUpdate.Entry.HotfixEntry");
    object entryInstance = Activator.CreateInstance(entryType);
    
    // 调用Start方法
    MethodInfo startMethod = entryType.GetMethod("Start");
    startMethod.Invoke(entryInstance, null);
}
```

## 📦 资源管理（Addressables）

### 资源 ≠ 代码

**重要**：Addressables管理的是**资源文件**，不是代码！

```
代码层级：
├── Framework层代码（C#）    → Assets/Scripts/Framework/
└── HotUpdate层代码（C#）    → Assets/Scripts/HotUpdate/

资源层级（Addressables）：
├── Framework资源分组         → 框架相关的资源文件
├── Common资源分组            → 通用资源文件
├── UI资源分组                → UI预制体、图集
└── Scene资源分组             → 场景文件
```

### Addressables分组说明

| 分组 | 存放内容 | 示例 |
|-----|---------|------|
| **Framework** | 框架相关的资源文件 | Loading界面、通用弹窗、默认字体 |
| **Common** | 游戏通用资源 | 通用特效、音效、图标 |
| **UI** | UI资源 | 各种UI预制体、UI图集 |
| **Scene** | 场景文件 | 游戏场景 |

## 🚀 启动流程

### 完整启动流程

```
1. Unity启动
   ↓
2. GameEntry.Awake()
   • 初始化所有Framework层Manager
   ↓
3. GameEntry.Start()
   • 检查本地版本
   • 连接更新服务器
   • 检查是否有更新
   ↓
4. 下载热更新资源（如果有更新）
   • 下载HotUpdate.dll
   • 下载Addressables资源包
   ↓
5. 加载HotUpdate.dll
   • HotUpdateManager.LoadHotUpdateAssembly()
   ↓
6. 加载AOT元数据
   • HotUpdateManager.LoadMetadata()
   ↓
7. 启动热更新逻辑
   • HotUpdateManager.StartHotfix()
   • 调用HotfixEntry.Start()
   ↓
8. 进入游戏逻辑（HotUpdate层）
```

### 代码示例

```csharp
// GameEntry.cs (Framework层)
private async void Start()
{
    // 1. 检查本地版本
    var localVersion = Framework.HotUpdate.VersionManager.GetLocalVersion();
    
    // 2. 检查更新
    string updateUrl = GetUpdateServerUrl();
    if (!string.IsNullOrEmpty(updateUrl))
    {
        var serverVersion = await HotUpdate.CheckUpdateAsync(updateUrl);
        
        // 3. 下载补丁
        if (serverVersion != null && serverVersion.Type == Framework.HotUpdate.UpdateType.HotUpdate)
        {
            await HotUpdate.DownloadPatchAsync(serverVersion);
        }
    }
    
    // 4. 加载热更新程序集
    HotUpdate.LoadHotUpdateAssembly();
    
    // 5. 加载元数据
    HotUpdate.LoadMetadata();
    
    // 6. 启动热更新逻辑
    HotUpdate.StartHotfix();
}
```

## 🔥 热更新流程

### 什么可以热更新？

✅ **可以热更新**：
- HotUpdate层的所有代码（HotUpdate.dll）
- Addressables管理的所有资源
- 配置表数据

❌ **不能热更新**：
- Framework层的代码
- Unity引擎代码
- 第三方插件代码（除非也在HotUpdate层）

### 热更新发布流程

```
开发阶段：
1. 修改HotUpdate层代码
2. 修改Addressables资源
3. 测试

发布热更新：
1. 编译HotUpdate.dll
2. 构建Addressables资源包
   • Build → New Build → Default Build Script
3. 上传到服务器
   • HotUpdate.dll.bytes
   • Addressables资源包（ServerData目录）
   • version.json（版本信息）
4. 玩家启动游戏时自动下载更新
```

## 💡 最佳实践

### 1. 代码分层原则

**Framework层应该放**：
- ✅ 稳定的底层服务
- ✅ 很少修改的代码
- ✅ 性能敏感的代码

**HotUpdate层应该放**：
- ✅ 游戏业务逻辑
- ✅ 经常变化的代码
- ✅ 需要快速迭代的功能

### 2. 资源管理原则

**Addressables应该放**：
- ✅ 预制体（Prefab）
- ✅ 音频文件
- ✅ 材质、贴图
- ✅ 场景文件
- ✅ 配置数据（JSON、XML等）

**不应该放Addressables**：
- ❌ C#代码文件（.cs）
- ❌ Assembly Definition文件（.asmdef）
- ❌ 编辑器脚本

### 3. 热更新策略

- 小更新：只更新HotUpdate.dll
- 中更新：更新HotUpdate.dll + 部分资源
- 大更新：更新所有资源
- 整包更新：修改了Framework层，需要重新发布安装包

---

## 🎯 总结

1. **Framework层** = 稳定的底层框架（不可热更新）
2. **HotUpdate层** = 游戏逻辑代码（可热更新）
3. **Addressables** = 资源文件管理（可热更新）
4. **三者独立** = 代码和资源分开管理

这样的架构设计既保证了框架的稳定性，又提供了灵活的热更新能力！🎉
